# YIAI Center V0.5.13 产品说明

> 文档版本：V0.5.13
> 创建日期：2026-07-21
> 更新日期：2026-07-22
> 产品范围：V0.5.10—V0.5.13 工单与无限轮人机共驾
> 正式地址：`http://192.168.50.112:19080`
> 当前 Active Release：`V0.5.13-unlimited-codrive`

## 1. 产品结论

本轮把 YIAI Center 从“能够解释工单流程”推进为“能够查询、确认并执行真实工单操作，同时支持员工与 AI 反复交接”的可演示产品。

核心规则如下：

- 工单能力必须在垂直 Agent 中装配，再随候选 Release 发布；工单页面本身不拥有 Agent 绑定权。
- 只读查询直接读取真实 SQLite 工单记录，不让模型编造状态或编号。
- 创建、更新、关闭和删除都先生成草稿，确认前不写数据库。
- 删除属于高风险操作，必须经过两次确认，最终执行软删除，审计记录永久保留。
- 人工承接没有轮次限制，也没有“完结会话”概念。
- 员工可以连续回复任意轮；只有点击“交还 AI”，AI 才重新取得输出权。
- AI 承接不表示问题完结。AI 始终处于待命状态，之后仍能再次进入人工协同。
- 同一时刻只允许 AI 或员工一方输出，避免双重答复。
- 每条新消息固定当时的 Active Release；历史 Run 保留原 Release、Agent、Tool、参数、结果和成本快照。

## 2. 角色与页面

### 2.1 用户页面

用户页面保留对话与历史消息，并新增：

- “我的工单”卡片，只展示固定演示用户自己的未删除工单。
- 写操作确认卡，展示 Tool、执行参数、执行前快照、剩余确认次数和操作结果。
- 人机共驾状态条，说明当前由 AI、等待员工还是员工承接。
- “邀请员工协助”入口，可在同一历史会话中发起人工协同。
- 每条 AI 消息下方的 Run 详情入口，用于查看 Router、Agent、Tool、模型调用和成本证据。

### 2.2 员工页面

员工页面是轻量的演示工作台，不建设完整坐席系统。它包含：

- 全部未删除工单卡片。
- 更新、关闭、软删除草稿入口。
- 待确认 Action 卡片及前后差异。
- 人机共驾卡片和交接包。
- 用户、AI、员工的完整消息时间线。
- 可连续发送的员工回复框。
- 唯一的状态交还操作“交还 AI”。

员工页面不提供“结束”“结案”或“完结共驾”按钮。工单自身可以关闭，但会话和 AI 待命状态没有完结概念。

### 2.3 平台管理页面

平台管理继续沿用 V0.5.9 的卡片式设计和 Agent 中心装配原则：

- Agent 卡片中配置 Skill、RAG、MCP Tool 和本轮新增的预置工单 Tool。
- 工单 Tool 以复选项显示读写属性、用途和输入结构。
- Release 卡片展示 Tool 新增与移除差异。
- 候选 Release 人工发布后，下一条新消息才使用新绑定。
- 新增 Agent 仍可独立创建、配置和发布，不依赖固定三个 Agent。

## 3. 版本能力

### V0.5.10：真实只读工单

新增两个只读 Tool：

- `list_work_orders`：按用户或员工视角读取工单列表。
- `get_work_order`：按工单编号读取完整详情。

用户查询工单时，Router 只选择一个 Agent；工单 Agent 根据 Release 中的 Tool 绑定执行查询。模型不可用时，系统仍可直接整理真实 Tool 结果，并在 Trace 中明确标记降级，不伪装成模型成功。

### V0.5.11：创建与确认

新增 `create_work_order`。自然语言先被整理为主题、描述、类别和优先级草稿。系统展示确认卡，用户确认后才由 Action Gateway 写入，并返回真实工单编号和操作收据。

每个 Action 有唯一幂等键。重复提交同一次确认只返回原结果，不重复创建工单。

### V0.5.12：更新、关闭与软删除

新增：

- `update_work_order`：展示字段差异，确认后更新允许字段。
- `close_work_order`：展示关闭结果，确认后把工单状态改为已关闭。
- `delete_work_order`：两次确认后写入 `deleted_at`，普通查询不再展示。

所有操作共用 Action Gateway、确认令牌、幂等规则、收据和追加式审计事件。

### V0.5.13：无限轮人机共驾

共驾状态只表示当前输出权，不表示事项生命周期：

`AI_ACTIVE → HANDOFF_REQUESTED → HUMAN_ACTIVE → AI_RESUMING → AI_ACTIVE`

产品负责人最终确认的规则已经覆盖旧版“一次人工回复”“一次交还”“结束会话”的设想：

- 员工回复次数不设上限。
- 用户在人工承接期间仍可补充消息。
- AI 在人工承接期间保持待命，但不输出。
- 员工点击“交还 AI”后，AI读取人工消息和处置摘要，再用当前 Active Release 继续承接。
- 模型调用失败也必须把输出权恢复给 AI，不能把会话卡死在中间状态。
- 交还之后可以再次邀请员工，循环次数不受限制。

## 4. 确认卡与操作反馈

确认卡用自然语言说明操作，而不是要求使用者理解内部状态码。至少展示：

- 操作类型与操作编号。
- 将要提交的参数。
- 执行前工单快照。
- 需要确认的次数。
- 当前是否仍未写入。
- 执行后的工单编号、结果和收据。
- 失败或结果未知时的原因。

一次性确认令牌只保存在当前浏览器，不进入 Run Trace、文档或数据库明文字段。数据库仅保存令牌哈希。

## 5. 运行证据与成本

工单查询、写操作和共驾交还都产生 Run。Trace 至少能还原：

`用户消息 → 固定 Release → Router → 垂直 Agent → Tool 或 Action → 结果 → 回答 → 最终状态`

本地 Tool、Action Gateway 和共驾状态迁移不产生模型 Token 成本。只有真实 DeepSeek 调用记录输入 Token、缓存输入 Token、输出 Token、价格快照和人民币估算成本。缺少 Provider 用量时保持未知，不补造数字。

## 6. 本轮明确不做

- 不建设登录、组织、权限或多租户系统。
- 不建设排队、抢单、坐席分配、服务时长或绩效系统。
- 不把工单词汇写入平台导航、Router 固定行业规则或通用数据结构名称。
- 不把 Agent 绑定入口迁回 Skill、RAG、MCP 或 Tool 自己的资源页面。
- 不把“工单关闭”等同于“会话结束”或“AI 不再待命”。
- 不用前端提示代替服务端确认、幂等和审计。

## 7. 验收口径

本版验收关注完整业务闭环：

- 只读查询能够返回真实记录并显示完整 Trace。
- 确认前数据库不变化，确认后只执行一次。
- 更新和关闭保存执行前后快照。
- 删除第一次确认不执行，第二次才软删除，审计仍可查。
- 人工可以连续回复至少三轮，状态始终为人工承接。
- 人工期间用户消息不触发 AI 回复。
- “交还 AI”后状态恢复为 AI 承接，模型用量如实记录。
- 同一会话可以再次进入人机共驾。
- 老 Release 和老 Run 不因新版本发布而被重写。
